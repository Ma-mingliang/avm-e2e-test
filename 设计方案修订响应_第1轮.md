# 设计方案修订响应 - 第1轮

## 问题 001 - 远程锁原子性不足

| 字段 | 内容 |
|------|------|
| 问题编号 | 001 |
| 是否接受 | 接受 |
| 修订位置 | git_ops.py, github_ops.py |
| 修订内容 | 远程锁改用 GitHub REST API `POST /repos/{owner}/{repo}/git/refs` 创建引用。该API在引用已存在时返回 422 Unprocessable Entity，实现真正的原子互斥。锁分支使用 `refs/heads/avm/system-lock`，锁数据存储在引用指向的 commit message 中。获取锁流程：1) 调用 API 创建引用；2) 成功则获取锁；3) 422 则锁已被占用；4) 其他错误则阻塞。释放锁流程：1) 删除引用 `DELETE /repos/{owner}/{repo}/git/refs/heads/avm/system-lock`。 |
| 验证方式 | 并发测试：两个进程同时调用 `create_lock_branch`，验证只有一个成功，另一个返回 422。 |
| 剩余风险 | GitHub API 限流可能导致锁获取失败（非竞争失败），需区分 422（锁被占）和 429（限流）。 |

## 问题 002 - 版本号计算可能遗漏远程未合并分支

| 字段 | 内容 |
|------|------|
| 问题编号 | 002 |
| 是否接受 | 接受 |
| 修订位置 | versioning.py |
| 修订内容 | 版本号计算必须从以下来源收集，任何来源查询失败必须阻塞版本预留：1) 本地标签 `git tag -l 'v[0-9]*'`；2) 远程标签 `git ls-remote --tags`；3) 远程分支名匹配 `agent/v[0-9]*-*`；4) 本地版本索引.json；5) 开放和关闭的PR标题/标签中的版本号（`gh pr list --json number,title,labels`）。任何 `gh` 命令失败（非零退出码）必须返回 `VERSION_CALC_BLOCKED` 错误，不得降级到仅本地计算。 |
| 验证方式 | mock `gh` 命令返回非零退出码，验证版本预留被阻塞并输出中文错误。 |
| 剩余风险 | 如果 GitHub 完全不可用，版本预留将被阻塞，用户需等待恢复。这是预期行为。 |

## 问题 003 - HMAC密钥存储位置未明确

| 字段 | 内容 |
|------|------|
| 问题编号 | 003 |
| 是否接受 | 接受 |
| 修订位置 | approval.py, config.py |
| 修订内容 | 使用 `keyring` 库存储HMAC密钥：1) 首次使用时生成 256-bit 随机密钥；2) 存储到 `keyring.set_password("avm", "approval-hmac-key", key_hex)`；3) Windows 上 keyring 默认使用 Windows Credential Manager；4) 读取失败时（如首次安装）自动生成并存储；5) 提供 `avm doctor --check-hmac-key` 命令验证密钥可用性；6) 提供 `avm rotate-hmac-key` 命令轮换密钥（旧签名失效）。不提供 fallback 到文件存储，因为文件存储安全性不足。 |
| 验证方式 | 在干净Windows环境测试：1) 首次生成密钥；2) 读取密钥；3) 重启后仍可读取；4) keyring不可用时的错误提示。 |
| 剩余风险 | 如果 Windows Credential Manager 被禁用或损坏，审批签名将不可用。`avm doctor` 会报告此问题。 |

## 问题 004 - 文档备份原子性在Windows上可能失败

| 字段 | 内容 |
|------|------|
| 问题编号 | 004 |
| 是否接受 | 接受 |
| 修订位置 | backup.py |
| 修订内容 | 1) 使用 `os.replace()` 替代 `os.rename()`，Windows 上 `os.replace` 是原子覆盖；2) 备份文件名格式 `{original_stem}_{doc-vN}_{timestamp}_{sha256_short8}{ext}`，SHA-256 前8位提供足够熵；3) 写入前检查目标是否存在，存在则比较 SHA-256，相同则跳过（幂等），不同则报错；4) 使用 `tempfile.NamedTemporaryFile(dir=target_dir)` 确保临时文件在同一文件系统。 |
| 验证方式 | 测试同名文件备份：第一次备份、第二次相同内容（跳过）、第二次不同内容（报错）。 |
| 剩余风险 | 极端情况下短哈希冲突（概率 1/2^32），可通过使用更长前缀缓解。 |

## 问题 011 - 文件锁在Windows重启后可能残留

| 字段 | 内容 |
|------|------|
| 问题编号 | 011 |
| 是否接受 | 接受 |
| 修订位置 | locking.py |
| 修订内容 | 锁文件设计：1) 锁文件格式 JSON，包含 `pid`、`hostname`、`created_at`、`expires_at`；2) 检测到锁文件时：a) 检查 `pid` 是否在当前机器运行（`os.kill(pid, 0)`）；b) 如果 PID 不存在且 `created_at` 超过 1 小时，自动清理并记录；c) 如果 PID 存在，锁有效；3) 远程锁分支检查本地锁文件一致性；4) `avm doctor --check-locks` 命令检查并报告残留锁。 |
| 验证方式 | 1) 创建锁文件，模拟进程崩溃（kill -9）；2) 重新运行，验证残留锁被检测和清理。 |
| 剩余风险 | 如果机器名相同且PID被复用（极低概率），可能误判。可通过增加随机nonce缓解。 |
