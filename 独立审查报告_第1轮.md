## 问题清单

### 问题 001 - 远程锁原子性不足
- **等级**：严重级
- **证据**：方案使用 `git push` 创建 `avm/system-lock` 分支作为远程锁，但 `git push` 不是原子操作。两个Agent可能同时检测到分支不存在，然后同时推送成功。
- **复现场景**：Agent A 和 Agent B 同时执行 `avm preflight`，都检测到无锁分支，都执行推送，Git服务器可能接受两次推送（第二次覆盖第一次）。
- **影响**：同项目可能有两个活动任务，违反 REQ-011。
- **必须修改**：使用 GitHub API 的 "create reference" 端点，该端点在引用已存在时返回错误，实现真正的原子互斥。或者使用 GitHub Actions 并发组。
- **验证方式**：编写并发测试，模拟两个进程同时获取锁。

### 问题 002 - 版本号计算可能遗漏远程未合并分支
- **等级**：严重级
- **证据**：版本号算法 `max(all_known_integer_versions) + 1` 依赖 Git 标签、版本索引.json、活动锁、远程任务分支和PR元数据。但如果Agent A预留v8后崩溃，未写入本地索引，远程分支存在但未被检测到（因为 gh 命令可能失败），Agent B可能计算出重复的v8。
- **复现场景**：网络短暂故障导致 `gh api` 失败，版本计算回退到仅本地标签。
- **影响**：版本号重复，违反 REQ-008。
- **必须修改**：版本计算必须包含本地标签、远程标签、远程分支名模式匹配、PR标题/标签中的版本号。任何来源查询失败必须阻塞版本预留，不得降级。
- **验证方式**：mock gh 命令失败，验证版本预留被阻塞。

### 问题 003 - HMAC密钥存储位置未明确
- **等级**：严重级
- **证据**：方案提到"建议使用全局工具私有密钥对审批记录做HMAC，密钥保存在Windows Credential Manager或受保护的全局配置区"，但未明确具体实现方式。
- **复现场景**：不同Windows版本的Credential Manager API不同，keyring库的行为可能不一致。
- **影响**：审批签名可能无法正常工作。
- **必须修改**：明确使用 `keyring` 库，指定 backend 为 `WindowsVaultKeyring`，并提供 fallback 到加密文件存储。密钥生成和轮换流程必须明确。
- **验证方式**：在干净Windows环境测试密钥存储和读取。

### 问题 004 - 文档备份原子性在Windows上可能失败
- **等级**：一般级
- **证据**：方案使用"临时文件 + 原子重命名"，但Windows上 `os.rename` 在目标已存在时会失败（不同于Unix的原子覆盖）。
- **复现场景**：备份目录已存在同名文件（加入短哈希可缓解，但极端情况仍可能冲突）。
- **影响**：备份可能不完整。
- **必须修改**：Windows上使用 `os.replace` 替代 `os.rename`，或先删除目标再重命名。备份文件名必须包含足够熵（如完整SHA-256前8位）。
- **验证方式**：测试同名文件备份场景。

### 问题 005 - CI配置可能被跳过
- **等级**：严重级
- **证据**：方案要求CI作为远程兜底，但GitHub Actions的 `workflow_dispatch` 允许手动触发跳过某些检查。如果Ruleset未正确配置"必需状态检查"，CI失败也不会阻止合并。
- **复现场景**：管理员在GitHub UI手动合并PR，绕过必需状态检查。
- **影响**：CI兜底失效。
- **必须修改**：初始化时必须检测并配置 Ruleset 的 `required_status_checks`，且 `strict: true`（要求分支最新）。工具必须验证配置生效。
- **验证方式**：创建测试仓库，验证CI失败时PR无法合并。

### 问题 006 - 审批失效检测时机不明确
- **等级**：一般级
- **证据**：方案说"审批后重新计算不一致则失效"，但未明确何时重新计算。是在 `avm publish` 时？还是在 `avm merge` 时？
- **复现场景**：用户在 `avm approve` 后、`avm publish` 前修改了文件，但工具未在合并前检查。
- **影响**：审批后变化的内容可能被发布。
- **必须修改**：在 `avm merge` 执行前必须重新计算 manifest_hash 并与审批记录比对。不一致则阻塞合并。
- **验证方式**：approve后修改文件，验证merge被阻塞。

### 问题 007 - 接手报告可能无限增长
- **等级**：一般级
- **证据**：方案要求"最新接手报告"在根目录，但未限制大小。如果项目有大量版本，报告可能包含过多历史索引。
- **复现场景**：项目经历100个版本后，接手报告超过100KB。
- **影响**：Agent加载报告时浪费上下文。
- **必须修改**：最新接手报告只保留最近5个版本的详细信息，更早的版本只保留索引链接。按需回查机制必须明确。
- **验证方式**：模拟大量版本，验证报告大小可控。

### 问题 008 - 废弃版本的PR保留可能泄露信息
- **等级**：建议级
- **证据**：方案要求废弃版本"保留PR作为远程证据"，但PR中可能包含敏感的提交信息或文件内容。
- **复现场景**：废弃任务的PR包含调试日志或临时凭据。
- **影响**：敏感信息在GitHub上可见。
- **必须修改**：废弃PR前必须扫描敏感信息。如果检出，必须先清理再关闭PR，或删除PR（但会丢失证据）。
- **验证方式**：废弃包含敏感信息的PR，验证被阻止或清理。

### 问题 009 - avm launch 可能无法正确注入环境变量
- **等级**：一般级
- **证据**：方案说"注入AVM环境变量"后启动Agent，但PowerShell的 `$env:` 设置只在当前进程有效。如果Agent是通过子进程启动的，环境变量可能丢失。
- **复现场景**：`avm launch claude` 启动 claude.exe，但 claude.exe 可能不继承 PowerShell 设置的环境变量。
- **影响**：Agent无法感知AVM上下文。
- **必须修改**：使用 `[Environment]::SetEnvironmentVariable` 设置用户级环境变量（持久化），或使用配置文件传递上下文。
- **验证方式**：启动Agent后检查环境变量是否可用。

### 问题 010 - 任务分类器对"修改Word的代码"判断可能误判
- **等级**：一般级
- **证据**：方案要求"修改处理Word的Python脚本"按文档任务询问，但分类器如何判断脚本的"最终目的"？如果脚本同时处理Word和Excel，如何分类？
- **复现场景**：用户要求"修改data_processor.py使其输出Word报告"，分类器可能只看到.py文件就判定为代码任务。
- **影响**：任务分类错误，可能绕过文档备份。
- **必须修改**：分类器必须分析任务描述（而非仅文件类型），使用关键词匹配（"Word"、"文档"、".docx"等）。存在歧义时必须询问用户。
- **验证方式**：测试各种任务描述，验证分类准确性。

### 问题 011 - 文件锁在Windows重启后可能残留
- **等级**：一般级
- **证据**：方案使用filelock库，但如果Agent进程崩溃或Windows重启，锁文件可能残留。
- **复现场景**：Agent在任务中间崩溃，锁文件留在磁盘。
- **影响**：后续任务被阻塞。
- **必须修改**：锁文件必须包含PID和创建时间。检测到残留锁时，检查PID是否仍在运行。如果PID不存在且锁超过一定时间（如1小时），自动清理并记录。
- **验证方式**：模拟进程崩溃，验证残留锁检测和清理。

### 问题 012 - 敏感信息扫描可能遗漏新型模式
- **等级**：建议级
- **证据**：方案要求扫描密钥、Token等，但未定义具体扫描规则。新类型的密钥（如AI API Key）可能不被识别。
- **复现场景**：用户提交包含 OpenAI API Key 的代码，但扫描器只检查 AWS/GitHub Token。
- **影响**：敏感信息泄露。
- **必须修改**：使用成熟的密钥扫描工具（如 truffleHog、detect-secrets），而非自定义正则。扫描规则必须可配置和更新。
- **验证方式**：测试各种类型的密钥，验证扫描覆盖率。
